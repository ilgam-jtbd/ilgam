# ADR-004 · 매칭 엔진 · 결제 · 알림 · API 계약

- **상태**: Accepted
- **일자**: 2026-04-23
- **참여 에이전트**: Backend, CX

## 매칭 엔진 (M1~M3)

**Postgres RPC(pl/pgSQL) + Edge Function 하이브리드.**

- 1차 필터(지역 반경·시급·시간대 overlap·차단 목록) = PostGIS `ST_DWithin` + 인덱스 RPC 10~50ms
- 2차 랭킹(거리·시급·평점·노쇼율 가중합) = pl/pgSQL
- **Claude API는 Edge Function(Deno)에서만** 공고 요약·금칙어 검사·CS 트리아지에 한정. 건당 $0.003~0.01
- 매칭 랭킹에 LLM 미사용 (레이턴시 1.5~3초 · 비용 부적합)
- M3 임계: 워커 1,500명·일 매칭 800건 초과 시 Python worker(임베딩 유사도) 분리 검토

## 당일급여 정산 · 법적 모델

**흐름**: 구인자 선결제(에스크로) → 워커 출근 GPS+셀카 인증 → 퇴근 인증 → 구인자 승인(15분 무응답 자동승인) → PortOne 송금 API 당일 지급.

**법적 지위**: **통신판매중개업 신고(자본금 요건 없음)**로 커버. 플랫폼 풀링은 전자금융거래법상 전자지급결제대행업 등록 의무(자본금 3억·금감원 신고) 발생 → M1~M2 부적합 → **PortOne 에스크로(가상계좌 예치)** 채택.

**당일정산 구조**: PortOne T+0 VIP 가맹점 계약 전까지 T+1이 현실. 이 경우 **운영자금 5천만~1억 확보**해 워커 당일 / 플랫폼 T+1 회수. 급구 당일급여 대비 동등 이상.

## 알림 어댑터 `notifyAdapter.send()`

- 시그니처: `send({templateId, channel, payload, userId})`
- **카카오 알림톡(Bizppurio) → 실패 시 SMS(알리고/LG U+) 폴백**
- 폴백 트리거: 알림톡 응답코드(수신거부·친구톡 차단·타임아웃). 평균 폴백 지연 8초 이내
- 큐잉: **pg-boss (Postgres 기반)** — Supabase 단일 DB 얹힘, 트랜잭션 일관성, 외부 의존 최소화
- 재시도: 지수 백오프 15s / 1m / 5m / 15m (총 4회). 최종 실패 dead-letter → 채널톡 CS 에스컬레이션
- 매칭 확정·급여 지급 등 critical은 우선순위 큐 분리

## Realtime vs 폴링

**하이브리드**.
- 구인자 앱: Supabase Realtime(Postgres CDC) — 지원·출근인증 실시간 푸시 (분 단위 경쟁)
- 시니어 워커 앱: **15초 간격 폴링 + Push Notification** 기본. 저가 안드로이드 배터리(웹소켓 시간당 3~5% 추가 소모)·LTE 단절 고려
- 매칭 수락 화면 등 단기 포그라운드(3분 이내)만 Realtime 채널 한시 구독 후 해제
- 동시 커넥션 워커 1,500명 전원 유지 대신 피크 200~300개로 절감

## API 계약

**Supabase RPC 직접 + tRPC 얇은 레이어.**
- Next.js·RN에서 `supabase-js` 공유 → M1 속도 최적
- 결제·정산·Claude 호출 등 서버 시크릿 필요 엔드포인트만 Next.js API Route 위에 tRPC 래핑
- zod 스키마 한 곳 통일, RN(Expo) 타입 추론 공유
- 읽기 = Supabase 직통, 쓰기·사이드이펙트 = tRPC 이원화
