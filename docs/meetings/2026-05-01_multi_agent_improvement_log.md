## Round 2 — UX/UI + PM: 핵심 화면 구현

### 결정 사항

**[UX/UI 에이전트 의견]**  
워커 앱 로그인이 없으면 모든 기능이 무의미. 시니어 UX 원칙 3가지:
1. 화면당 CTA 1개
2. 최소 글자 크기 18pt (시스템 200% 대응)
3. 입력 필드 최소 높이 56dp

**[PM 에이전트 반론]**  
구인자 승인 UI가 더 급선무. 워커가 지원해도 구인자가 수락 못 하면 매칭이 0. B2B 구인자가 먼저 확보돼야 워커 모집 의미.

**[오케스트레이터 결론]**  
둘 다 병렬 구현. 의존 관계 없음.

### 구현 결과

- `apps/admin-web/app/(dashboard)/applicants/page.tsx` — 지원자 목록 RSC
- `apps/admin-web/app/(dashboard)/applicants/ApplicantActions.tsx` — 1탭 승인/반려 클라이언트 컴포넌트
- `apps/worker-app/app/index.tsx` — OTP SMS 로그인 (Supabase `signInWithOtp`)

## Round 3 — BM: 수익 대시보드

### 에이전트 피드백

**[BM 에이전트]**  
GMV와 수수료를 별도 지표로 보여줘야 함. 투자자에게 "GMV vs 수수료 수익" 혼동이 없어야 IRR 설명 가능. 30일 윈도우로 현황 파악.

**[IR 에이전트]**  
대시보드 숫자가 곧 IR 덱 업데이트 소스. 타이미 FY25 GMV 342억 엔 대비 일감 GMV를 같은 기준으로 보여줘야 투자자 설득 가능.

**[QA 에이전트]**  
`Promise.all` 집계 시 하나라도 실패하면 전체 실패. 각 쿼리를 독립적으로 처리해야 대시보드가 부분 로드됨.

**[오케스트레이터 결론]**  
QA 의견 반영: 각 집계 독립 처리, 실패 시 0으로 대체.

### 구현 결과

- `apps/admin-web/app/(dashboard)/page.tsx` — GMV·수수료·매칭·공고·지원자·워커 6종 지표 카드

## Round 4 — QA: 자동화 테스트

### 에이전트 피드백

**[QA 에이전트]**  
1인 운영에서 테스트 없는 배포는 러시안 룰렛. 최저 안전망:
1. 스키마 경계값 (최저임금, 전화번호 형식, 위경도 범위)
2. 매칭 필터 (자격증 필터, 노쇼 차단)
3. 결제 HMAC (Deno 환경이라 vitest 직접 테스트 불가 → pgTAP으로 대체)

**[백엔드 에이전트]**  
`PhoneE164` 정규식이 `+8201012345678` (올바르지 않은 형식)을 허용하지 않음이 코드상 의도. 테스트에서 이를 "통과해야 함"으로 잘못 작성 → 수정 필요.

**[오케스트레이터 결론]**  
백엔드 에이전트 의견 즉시 반영. 테스트는 코드의 거울, 틀린 기대값은 버그.

### 구현 결과

- `packages/core/src/schemas.test.ts` — 13개 단위 테스트 (PhoneE164·JobCreate·WorkerPreferences·ClockIn)
- 실패 케이스 수정: `+8201012345678` → `+821112345678` (011 번호)

## 에이전트 토의 핵심 교훈

1. **BM vs PM 충돌** → 병렬 구현으로 해소. 의존 없으면 순서 논쟁 불필요.
2. **QA의 역할** → 구현 후 검증이 아닌 설계 단계에서 경계값 명세 제공.
3. **UX의 역할** → 시니어 UX 원칙(56dp, 18pt, 단일 CTA)이 모든 화면에 일관 적용.
4. **IR의 역할** → 코드 지표가 곧 IR 수치. 대시보드 설계 시 투자자 시각 반영 필수.
5. **오케스트레이터 원칙** → 에이전트 간 의견 충돌 시 "데이터·시간·범위" 3축으로 판단.
